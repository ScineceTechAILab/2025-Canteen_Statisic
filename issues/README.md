这是一个很常见的开发现象，通常称为**“回归测试” **或** “缺陷跟踪”**。建议你在项目根目录下建立一个专门的**缺陷记录与回归测试** 文件夹，比如叫 `issues` 或 `bug_reports`，并配合规范的文档记录和自动化测试。

### 推荐做法

#### 1. 建立文件夹

在你的项目根目录下新建一个文件夹，例如：

**/issues**

或

**/bug_reports**

#### 2. 记录格式

每次发现新 bug 或修复 bug 时，在该文件夹下新建一个 Markdown 文件，命名建议用日期+简要描述，例如：

**2025-07-04-fix-export-bug.md**

内容可以包括：

* **Bug描述** ：出现的现象、报错信息、截图
* **复现步骤** ：如何触发
* **修复方案** ：你做了哪些修改
* **修复后新发现的问题** ：如果有，继续记录

#### 3. 自动化测试

建议你在 `tests/` 文件夹下为每个 bug 写一个最小复现的单元测试，修复后保证测试通过，避免回归。

#### 4. 版本管理

* 每次修复 bug 后， **提交到 git 并写明 commit message** ，如 `fix: 修复主表页计写入异常`
* 可以在 `issues/README.md` 里维护一个 bug 列表和修复进度

#### 5. 参考结构

**/issues**

**    /2025-07-04-fix-export-bug.md**

**    /2025-07-04-fix-sheetname-bug.md**

**/tests**

**    /test_export_bug.py**

**    /test_sheetname_bug.py**

---

**总结：**

* 用 `issues/` 或 `bug_reports/` 文件夹记录每次 bug 及修复过程
* 配合单元测试，防止回归
* 结合 git 版本管理，便于追踪和回滚

这样可以让你的开发过程更有条理，bug 修复历史清晰可查，团队协作也更高效
